# CONCEPT · 03

Shebang — #!/bin/bash

스크립트 첫 줄에 적는 마법의 두 글자 #! — "이 파일은 누구한테 읽혀야 하나요?"의 답입니다.

두 줄짜리 스크립트의 골격
# hello.sh
#!/bin/bash
echo "Hello, $USER"
  • 1행 #!/bin/bash = Shebang (sha-bang)
  • "이 스크립트는 /bin/bash로 실행해라"라는 표시
  • 나머지 줄은 그 쉘이 해석하는 명령
Shebang을 빼면?
# hello.sh (Shebang 없음)
echo "Hello, $USER"
→ Ubuntu에서 ./hello.sh 실행 시 dash로 실행됨
→ bash 전용 문법 (배열 등)은 동작 X
  • 없어도 bash hello.sh는 동작
  • ./hello.sh은 시스템 기본 쉘로 실행
  • 이식성·예측가능성을 위해 반드시 명시 권장

#! 다음의 경로는 반드시 절대경로 — /bin/bash·/usr/bin/env bash 같이 쓰며, 공백·줄바꿈 없이 첫 줄에 와야 합니다.

실행 방법 두 가지 — ./script.sh vs bash script.sh

같은 결과처럼 보여도 권한 요구와 사용 쉘이 다릅니다.

./hello.sh
파일을 직접 실행 (Shebang으로 쉘 결정)
1. chmod +x hello.sh # 실행 권한 부여
2. ./hello.sh # 현재 디렉터리의 hello.sh 실행
! 권한 없으면 Permission denied
장점실제 명령처럼 동작 · Shebang 존중 · PATH 등록도 가능
주의매번 chmod +x 필요
bash hello.sh
bash에게 "이 파일 읽어줘" 요청
1. bash hello.sh # 권한 없어도 즉시 실행
2. # Shebang 무시 — 무조건 bash로 동작
! 읽기 권한(r)만 있으면 OK
장점권한 부여 불필요 · 디버깅 빠름 · bash -x로 추적 가능
주의Shebang을 무시함 — 휴대성 ↓

실무에서는 ./script.sh 방식으로 "명령어처럼 쓰는" 것이 일반적 — Shebang을 정확히 적고 권한을 부여하는 게 좋은 습관입니다.

변수 = "이름표 붙은 상자"

대입은 =로, 참조는 $로 — bash는 이 두 글자가 거의 전부입니다.

올바른 사용
name="홍길동" # 대입
age=20 # 숫자도 그냥 문자열
echo "안녕 $name" # 참조 → 안녕 홍길동
echo "${name}_log.txt" # 중괄호로 경계 표시
  • = 양옆에 공백 없음
  • 문자열은 "..."로 감싸기 (변수 치환 O)
  • 변수명+문자가 붙으면 ${var}로 경계 표시
자주 하는 실수
name = "홍길동" # # command not found
echo '안녕 $name' # # 안녕 $name (치환 X)
echo $name_log # # 변수명을 name_log로 오해
name=홍 길동 # # 길동: command not found
  • = 옆에 공백 → 명령으로 인식
  • '..' 작은따옴표 안의 $는 치환 안 됨
  • 값에 공백이 있으면 반드시 "..."

bash 변수는 모두 문자열 — 산술은 $(( ))로 따로 다룹니다. (예: echo $((age + 1)))

환경 변수와 export — 자식에게도 알리려면

변수에는 이 쉘만 아는 것(지역)과 자식 프로세스에도 전달되는 것(환경) 두 종류가 있습니다.

지역 변수 (Local)
현재 쉘에서만 유효 — 자식엔 안 보임
MYVAR="hello"
echo $MYVAR # hello
bash # 자식 쉘 진입
echo $MYVAR # (빈 줄) — 안 보임
  • 대부분의 스크립트 변수는 지역
  • 쉘 종료하면 사라짐
  • 자식 프로세스에 전달 X
환경 변수 (Environment)
export로 자식 프로세스에도 전달
export MYVAR="hello"
bash # 자식 쉘 진입
echo $MYVAR # hello — 보임 ★
env | grep MYVAR # 환경 변수 목록 확인
  • PATH·HOME·USER·SHELL도 모두 환경 변수
  • env·printenv로 목록 조회
  • 로그인 시 ~/.bashrc·~/.profile에서 설정

"자식 프로세스"는 곧 우리가 실행하는 스크립트·프로그램 그 자체 — 스크립트가 변수를 읽으려면 export가 필요합니다.

# backup.sh
#!/bin/bash
set -euo pipefail

SRC="/var/log"
DST="$HOME/backup"
TODAY=$(date +%F)
FILE="$DST/log-$TODAY.tar.gz"

mkdir -p "$DST"
tar -czf "$FILE" $SRC 2>/dev/null || true

# 7일 지난 파일 삭제 (개수 카운트)
REMOVED=$(find "$DST" -name "log-*.tar.gz" -mtime +7 -print -delete | wc -l)

echo "[$(date '+%F %T')] 백업 완료 → $FILE"
echo "오래된 백업 $REMOVED개 삭제"

Git의 3영역 — Working · Staging · Repository

파일이 거쳐가는 세 개의 방 — 이 그림 하나가 Git의 전부라 해도 과언이 아닙니다.

1. Working
작업 디렉터리 — 우리가 편집

에디터로 열어 글자를 치고 저장하는 그 폴더 자체. 변경은 여기서 일어남.

  • 실제 파일이 살고 있는 곳
  • ls·cat으로 그대로 보임
  • 아직 Git이 추적 안 한 상태

2. Staging
준비 영역 — 무대(Index)

다음 커밋에 포함할 변경을 "골라 담는" 공간. 사진 찍기 직전의 포즈.

  • 커밋 후보를 모으는 임시 영역
  • git status로 확인
  • git reset HEAD 파일로 빼기
git commit
→
3. Repository
저장소 — 영구 보관

커밋되어 영구히 남는 스냅샷의 모음. .git/ 폴더에 보관.

  • 되돌릴 수 있는 모든 시점
  • 각 커밋은 고유 해시(예: a3f9e1)
  • git log로 히스토리 확인

편집 → git add(무대 올리기) → git commit(사진 찍기) — 이 순환만 외우면 됩니다.

# COMMAND · 03

git add & git commit — 무대에 올리고 사진 찍기

변경을 골라 담고 영구히 기록하는 두 단계 — Git에서 가장 자주 칠 명령어들입니다.

git add — 무대에 올리기
Working → Staging
# 파일 하나만 / 여러 개 / 전체
git add README.md # 한 파일
git add *.md # 패턴 매칭
git add . # 현재 폴더 전체
git add -A # 삭제 포함 모든 변경
  • 이 단계에서 "어떤 변경을 같이 묶을지" 선택
  • 실수로 올렸으면 git restore --staged 파일
  • .gitignore에 등록된 파일은 무시
git commit — 사진 찍기
Staging → Repository
# 메시지와 함께 기록
git commit -m "feat: README 작성"
# -m 없이 치면 에디터가 열림

# 좋은 메시지 예
"fix: 로그인 시 빈 비번 허용 버그 수정"
"docs: 설치 가이드에 OS별 분기 추가"
  • 커밋은 "왜 바꿨는가"를 적는 짧은 편지
  • 한 커밋 = 한 가지 의미 단위로 작게
  • feat / fix / docs / refactor 접두어 권장

"add로 골라 담고 commit으로 도장 찍기" — 이 두 단계가 분리돼 있어서 관련된 변경만 한 커밋에 묶을 수 있습니다.

# CONCEPT · 03

병합의 두 가지 — Fast-forward vs 3-way merge

main이 가만히 있었나, 아니면 같이 움직였나 — 그 차이만 알면 됩니다.

Fast-forward
main이 가만히 있을 때 — 포인터만 이동
  • 분기 후 main에 새 커밋이 없음
  • 병합 커밋 없이 main 포인터만 앞으로
  • 히스토리가 일직선 — 깔끔
3-way Merge
main도 따로 움직였을 때 — 합치는 커밋 생성
  • 두 부모를 가진 merge commit 생성
  • 병합 시 자동 메시지 또는 직접 작성
  • 여기서 충돌이 발생할 수 있음

같은 줄을 양쪽에서 수정했을 때만 충돌이 발생 — 다른 줄·다른 파일이면 Git이 알아서 합쳐줍니다.

# CONCEPT · 02

reset vs revert — 히스토리 되감기 vs 취소 커밋

"없던 일로" vs "되돌렸다는 기록을 남기고 취소" — 협업 중엔 후자가 안전합니다.

git reset
최근 커밋을 지움 — 히스토리 자체를 수정
git reset --soft HEAD~1 # 변경 보존
git reset --mixed HEAD~1 # 기본값
git reset --hard HEAD~1 # 파일까지 삭제 ⚠
  • 로컬에만 있는 커밋 정리에 적합
  • 이미 push된 커밋엔 절대 금지
git revert
"이 커밋을 되돌리는" 새 커밋 추가
git revert HEAD # 직전 커밋 취소
git revert <커밋해시> # 임의 커밋 취소
# → 자동으로 "Revert ..." 커밋 생성
  • 히스토리를 안전하게 유지
  • push 한 커밋 되돌릴 때 정답

황금률: push된 커밋은 reset 금지 → revert로 되돌리기. reset은 내 로컬에서만, 그것도 --hard는 최후의 수단입니다.
