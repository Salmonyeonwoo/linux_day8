# CONCEPT · 02

프로세스의 번호표와 상태

모든 프로세스는 고유 번호 (PID)와 부모 번호 (PPID)를 가지며, 매 순간 어떤 상태에 있습니다.

PID · PPID — 부모-자식 관계

PID = 나의 주민번호 · PPID = 나를 띄운 부모의 PID

student@linux: ~

$ ps -o pid,ppid,comm

  PID  PPID  COMMAND

 2841  2840  bash   # 내 셸

 3105  2841  ps     # bash가 띄운 자식

ps의 PPID(2841)가 bash의 PID(2841)와 똑같습니다 — "bash가 ps를 띄운 부모" 라는 뜻. 이 관계가 모이면 프로세스 트리가 됩니다 (다음 시간에 pstree로 확인).

# COMMAND

ps

현재 시스템의 프로세스를 스냅샷으로 한 번 찍어 보여주는 명령. 가장 기본적인 관찰 도구입니다.

-e 모든 프로세스 표시 (자기 셸 외 전부)
-f 풀 포맷 (UID · PID · PPID · STIME · CMD)
-u 지정한 사용자의 프로세스만
-o 원하는 컬럼만 골라 출력 (예: pid,ppid,comm)
aux BSD 스타일 묶음 — %CPU · %MEM · STAT · RSS 포함
$USER 는 현재 로그인 사용자 이름이 자동으로 들어 있는 셸 변수입니다. 따옴표 없이 그대로 명령 인자로 넘길 수 있어요 (예: echo $USER).
student@linux: ~

# 1. 그냥 ps — 현재 셸의 프로세스만
$ ps

# 2. 모든 프로세스를 풀 포맷으로
$ ps -ef

# 3. BSD 스타일 — 자원 사용량까지
$ ps aux

# 4. 필요한 컬럼만 골라서
$ ps -o pid,ppid,comm

Tip: ps에는 두 스타일이 공존합니다 — -ef(SysV, 하이픈 사용) 과 aux(BSD, 하이픈 없음). 둘 다 자주 쓰니 알아두세요.

조건

# 결과 확인 — 헤더 한 줄을 읽으면 됨
출력 첫 줄(컬럼 헤더)에 아래 4개가 모두 보이는가?
%CPU · %MEM · STAT · RSS
→ 출력이 길면 첫 줄만 확인하면 충분

# COMMAND

pstree

ps가 목록이라면 pstree는 가계도. 부모-자식 관계를 한눈에 트리 모양으로 보여줍니다.

-p 각 프로세스의 PID 같이 표시
-a 실행 시 넘긴 명령행 인자(arguments)도 표시
-u 자식의 사용자가 부모와 다르면 사용자명 표시
-T 스레드는 숨기고 프로세스만 (출력이 깔끔)
USER · PID 위치 인자: 특정 사용자 / PID / 명령어부터 보기
$$ 는 현재 셸의 PID가 들어 있는 셸 변수입니다. echo $$ 또는 pstree $$ 처럼 그대로 인자로 넘길 수 있어요.
student@linux: ~

# 1. 시스템 전체 트리
$ pstree

# 2. PID까지 같이
$ pstree -p

# 3. 특정 명령어/사용자만 (위치 인자)
$ pstree -p bash

# 출력 형태 (간략 예)
systemd(1)─┬─sshd(820)─┬─bash(2841)───ps(3105)
          └─systemd-journal(415)

Tip: 출력이 길면 | less 로 페이지 단위 보기. 최상단(PID 1)에는 보통 systemd 또는 init이 있어요.

실습 — pstree로 가계도 그리기

앞 슬라이드의 옵션과 셸 변수를 응용해 부모-자식 관계를 직접 추적합니다.

01
전체 트리를 PID와 함께
★☆☆

시스템의 모든 프로세스를 PID가 함께 보이도록 트리 형태로 출력하시오.

조건
  • 트리 모양으로 들여쓰기되어 출력될 것
  • 각 프로세스 옆에 PID 숫자가 보일 것 — 옵션 한 글자
  • 출력이 길면 | less 로 페이지 단위 보기 (스페이스: 다음 페이지 · q: 종료)
  • 가장 위(맨 첫 줄)에 있는 프로세스의 PID와 이름을 메모해 두기
# 결과 확인 — 보조 명령으로 교차 검증
$ ps -p 1 -o pid,comm
→ 트리 최상단에 본 이름과 같은가? (보통 systemd 또는 init)
02
내 셸의 부모를 찾아라
★★☆

지금 사용 중인 내 셸의 부모 프로세스 이름을 알아내시오. (sshd, login, gnome-terminal 등 — 환경에 따라 다름)

조건
  • 자기 셸 PID를 직접 타이핑하지 말고 셸 변수 활용 (앞 슬라이드 인포박스 참고)
  • pstree에 그 변수를 위치 인자로 넘기면 자기부터 위 부모 체인이 보임
  • 출력의 가장 윗줄에서 bash 바로 위쪽 노드가 부모
# 결과 확인 — 다른 경로로 교차 검증
$ ps -o ppid= -p $$ → 부모 PID 한 줄
$ ps -o comm= -p <위 PID> → 부모 이름
→ pstree 결과의 부모 이름과 일치하는가?

★☆☆

02

★★☆

조건

# CONCEPT · 02

Job ID vs PID — 작업의 두 이름

하나의 프로세스가 두 가지 번호로 불립니다 — 누가 매겼느냐로 갈립니다.

같은 프로세스, 두 이름

[1] 은 셸이 매긴 작업 번호 · 3421 은 커널이 매긴 PID

student@linux: ~

$ sleep 100 &  # 백그라운드로 시작

[1] 3421    # 셸이 알려주는 두 번호

$ jobs

[1]+ Running   sleep 100 &  # 셸의 작업 목록 — Job ID 사용

$ ps -p 3421  # 커널 도구 — PID 사용

[1] 과 3421 은 완전히 같은 프로세스입니다. 작업 제어 명령(jobs/fg/bg)에는 %1, 시스템 도구(ps/top/kill)에는 3421 — 도구가 다를 뿐.

두 번호의 차이

%N

Job ID

셸이 매긴 작업 번호

  • 표기: %1, %2, %+(현재), %-(이전)
  • 한 셸 안에서만 유효 — 셸 닫으면 사라짐
  • 인자로 받는 명령: jobs · fg · bg
PID

Process ID

커널이 매긴 시스템 전체 번호

  • 표기: 3421, 2841 (1교시에서 봤던 그 PID)
  • 시스템 전체 고유 — 어느 셸에서든 통함
  • 인자로 받는 명령: ps · top · kill

백그라운드로 보내기

두 가지 경로 — 처음부터 뒤로 보내거나, 도중에 정지시켜 뒤로 보내거나.

& 명령어 끝에 붙여 처음부터 백그라운드 실행 기호 · 입력 시
Ctrl+Z 도중에 일시정지 → Stopped 상태로 백그라운드 이동 단축키 · 실행 중

[N] PID 셸이 알려주는 출력 — [Job ID] PID 셸 출력

작업 다루기

만들어 놓은 작업을 보고 · 가져오고 · 다시 보내는 세 가지 동사.

jobs 현재 셸의 작업 목록 — Job ID·상태·명령 보기

student@linux: ~

$ sleep 100 &   $ sleep 200

[1] 3421    # 진행 중 ... Ctrl+Z

[2]+ Stopped   sleep 200

$ jobs

[1]-  Running    sleep 100 &

[2]+ Stopped    sleep 200

$ bg %2   # 정지된 200을 뒤에서 계속

[2]+ sleep 200 &

$ fg %1   # 작업 1을 앞으로

sleep 100   # 이제 셸은 sleep 끝까지 대기

Tip: fg 는 가장 자주 쓰는 단축 — 인자 없이 치면 + 작업이 앞으로 옵니다. 작업이 정지(Stopped) 상태일 때만 bg 가 의미 있음.

# COMMAND

영속화 — nohup · disown

터미널을 닫아도 죽지 않는 작업 만들기. 처음부터(nohup) 와 도중에(disown) 두 방법.

nohup CMD & 처음부터 SIGHUP 무시 + 백그라운드 시작 시
disown %N 이미 도는 작업을 셸 작업 목록에서 떼어냄 도중에
disown -h %N 목록은 유지하되 SIGHUP만 차단 중간 옵션
nohup.out nohup 작업의 stdout이 자동 저장되는 파일 (현재 폴더) 자동 생성
SIGHUP 은 터미널이 닫힐 때 셸이 자식에게 보내는 "나 끝낸다" 신호 — 받으면 자식도 같이 죽음. nohup은 이걸 처음부터 무시하게 만들고, disown은 셸이 추적을 멈춰 자식에게 안 보내게 합니다.
student@linux: ~

SCENE 1 # 처음부터 nohup

$ nohup ./long-task.sh &

[1] 3421

nohup: ignoring input and appending output to 'nohup.out'

$ exit  # 이 터미널 닫아도 3421은 살아남음

SCENE 2 # 이미 도는 작업 떼어내기

$ sleep 1000 &

[1] 3422

$ disown %1

$ jobs  # 더 이상 목록에 없음

$ # 이제 exit 해도 3422는 계속 살아 있음

Tip: 적용 시점이 다릅니다 — nohup 은 실행 전, disown 은 실행 후. 살아남은 프로세스는 부모가 init/systemd (PID 1) 로 바뀝니다.

# CONCEPT · 02

주요 시그널 5종

전체는 64개지만 5개 면 99% 상황은 해결됩니다.

번호 vs 이름 — 둘 다 통합니다

kill의 인자로 번호 또는 이름 모두 받습니다.

student@linux: ~

$ kill -9 3421  # 번호로

$ kill -KILL 3421  # 이름으로 — 같은 효과

$ kill -SIGKILL 3421  # SIG 접두사도 OK

$ kill -l  # 전체 시그널 목록

1) SIGHUP  2) SIGINT  3) SIGQUIT  9) SIGKILL

15) SIGTERM 18) SIGCONT 19) SIGSTOP ... (총 64개)

kill 의 기본값은 SIGTERM (15) — 인자 없이 kill PID 만 쳐도 친절한 종료 요청. -9 는 "정말 안 죽을 때" 만 사용.

외워둘 5종

1
SIGHUP — Hangup
터미널 끊김. 데몬은 설정 재로드 신호로 활용
2
SIGINT — Interrupt
Ctrl+C 의 정체. 사용자 인터럽트
9
SIGKILL — Force kill
무시 불가. 핸들러 등록 불가 — 최후 수단
15
SIGTERM — Terminate
기본값. 정리 시간을 주는 친절한 종료 요청
19/18
SIGSTOP / SIGCONT
정지 / 재개. Ctrl+Z + bg 와 같은 효과

# COMMAND

kill

이름과 달리 "죽이는 명령" 이 아니라 "시그널 보내는 명령". 보내는 시그널에 따라 결과가 달라집니다.

kill PID 기본 — SIGTERM(15) 보내기 (친절한 종료) 기본값
kill -9 PID SIGKILL — 무시 불가 강제 종료 최후 수단
kill -NAME PID 이름으로 지정 (-HUP, -TERM 등) 이름 표기
kill -l 시그널 전체 목록 (1~64) 참고
kill %N Job ID로도 보내기 가능 (2교시 작업 제어와 연결) 셸 통합
같은 시그널을 보내는 3가지 표기 — -9, -KILL, -SIGKILL 모두 동일합니다. 외울 때는 번호 가 짧아 편합니다.

# COMMAND

이름으로 종료

PID를 일일이 찾기 귀찮을 때 — 이름 패턴으로 한꺼번에.

pgrep PAT 패턴 매치 PID 미리보기 (실제로 안 죽임)
pkill PAT 패턴 매치 프로세스에 SIGTERM
pkill -9 PAT 패턴 매치 프로세스에 SIGKILL 강제
killall NAME 정확한 이름 매치 — 동명 프로세스 전부 종료
pkill -u USER 특정 사용자의 프로세스만 대상으로
pkill 은 부분 일치(정규식), killall 은 정확 일치. pkill chro 는 chrome도 잡지만 killall chro 는 못 잡습니다. 실수 방지로 pgrep 먼저 권장.
student@linux: ~

$ sleep 100 & sleep 200 & sleep 300 &

# 1. 누가 잡힐지 미리 보기
$ pgrep sleep

3421
3422
3423

# 2. 확인됐으니 종료
$ pkill sleep

[1] Terminated sleep 100 &
[2]- Terminated sleep 200 &
[3]+ Terminated sleep 300 &

# 3. 강제 + 사용자 한정
$ pkill -9 -u $USER firefox

Tip: 안전한 흐름 — pgrep 로 누가 잡힐지 확인 → 같은 패턴을 pkill 에 넘기기. root 권한일 때 패턴 잘못 쓰면 시스템 프로세스도 같이 날아갑니다.

After=network.target

ExecStart=/usr/local/bin/myapp.sh

Restart=on-failure

[Install]

WantedBy=multi-user.target

파일 위치는 /etc/systemd/system/ (관리자 추가) 또는 /usr/lib/systemd/system/ (패키지 기본). 작성 후 systemctl daemon-reload 필수.

[Service]

[Install]

# COMMAND

oneshot & 의존성

"언제 시작" 과 "누구 다음" — 의존성 지시자.

Type=oneshot 한 번 실행 후 종료 (성공 = 정상)
RemainAfterExit=yes 종료해도 "active" 로 유지 (의존 서비스 만족)
After=NAME NAME 이후에 시작 — 순서만
Requires=NAME NAME 필요 — NAME 실패 시 같이 실패
Wants=NAME NAME 권장 — 실패해도 계속 진행
After 와 Requires 는 자주 함께 — Requires(=의존) + After(=순서). 둘 중 하나만으로는 부족.
initdb.service — 초기화용 oneshot

# /etc/systemd/system/initdb.service

[Unit]

Description=DB 초기 데이터 적재 (1회)

After=network.target postgresql.service

Requires=postgresql.service

[Service]

Type=oneshot

ExecStart=/usr/local/bin/initdb.sh

RemainAfterExit=yes

[Install]

WantedBy=multi-user.target

Tip: 이 유닛을 enable 하면 부팅 시 postgresql 이 켜진 뒤에야 initdb 가 한 번 실행됨. RemainAfterExit=yes 덕에 "init이 끝났음"이 상태로 남아 다른 서비스가 의존 가능.

/etc/logrotate.d/ 서비스별 회전 정책 (앱당 한 파일)

/etc/logrotate.d/myapp — 회전 정책 예시

/var/log/myapp.log {

    daily   # 매일 회전

    rotate 7   # 7개 보관 후 폐기

    compress   # 오래된 것은 gzip

    missingok   # 파일 없어도 오류 안

    notifempty   # 빈 파일은 회전 안

    create 0640 myapp myapp

    postrotate

       systemctl reload myapp

    endscript

}

Tip: sudo logrotate -d /etc/logrotate.d/myapp 로 드라이런 (실제 회전 없이 시뮬레이션). -f 는 강제 실행.

# SCENARIO · 01

시나리오 ① — 내 스크립트를 정식 서비스로

5교시 service unit + 4교시 enable + 6교시 journalctl 한 흐름.

상황 — 1분마다 디스크 사용량을 /tmp/disk.log 에 기록하는 모니터링 스크립트를 서비스로 정식 등록하고, 죽으면 자동 재시작 + 로그 실시간 확인.

EDIT
vim /usr/local/bin/disk-monitor.sh  ·  vim /etc/systemd/system/disk-monitor.service
스크립트(while true; df -h >> /tmp/disk.log; sleep 60; done) + 유닛 작성. Restart=on-failure 포함.
RUN
sudo systemctl daemon-reload && sudo systemctl start disk-monitor
유닛 인식 후 시작. daemon-reload 잊지 말 것.
CHECK
systemctl status disk-monitor  ·  systemctl is-active disk-monitor
Active: running 확인 + Main PID 메모.
ENABLE
sudo systemctl enable disk-monitor
부팅 자동 시작 등록.
WATCH
sudo kill -9 <MainPID>  ·  journalctl -u disk-monitor -f
강제 종료 시뮬 → 실시간 로그에서 자동 재시작과 새 PID 확인.

# SCENARIO · 02

시나리오 ② — 장애 대응 흐름

"사이트가 안 떠요" — 어디부터 봐야 할까?

상황 — 사용자 신고: "웹사이트가 안 떠요." 서버에 SSH 접속해서 1교시~6교시 도구를 순차 적용해 원인 파악 + 복구.

DETECT
systemctl status nginx
서비스 죽었는지부터 — Active 가 failed/inactive 인지 확인. 4교시.

# TROUBLESHOOTING

증상 → 첫 손길

자주 만나는 증상별 대응 1·2단계.

"서비스가 안 떠요"

  • systemctl status NAME — 상태와 최근 10줄 로그
  • journalctl -u NAME -xe — 자세한 에러
  • 유닛 수정한 적 있으면 daemon-reload 확인
  • ExecStart 경로/권한 확인 — 가장 흔한 원인

"시스템이 느려요"

  • top 또는 htop — P / M 정렬
  • 1위 프로세스의 RES와 %CPU 메모
  • 필요 시 renice -n 15 -p PID 로 양보 또는 종료
  • df -h — 디스크 가득 차지 않았나

"프로세스가 안 죽어요"

  • 먼저 kill PID (TERM)
  • 20초 기다려도 살아 있으면 kill -9 PID
  • ps -o stat,comm 으로 STAT 확인 — Z 면 좀비 (부모를 죽여야)
  • D 면 디스크 I/O 대기 — 종료 어려움, 재부팅 검토

"부팅이 느려요"

  • systemd-analyze — 총 시간
  • blame 상위 3개가 범인 후보
  • critical-chain 으로 의존 체인 확인
  • 불필요 서비스는 disable 또는 mask
