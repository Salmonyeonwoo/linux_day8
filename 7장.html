# CONCEPT · 01

컴퓨터 안의 "서랍장" — 저장장치란?

전원을 꺼도 데이터가 사라지지 않는 칸들 — 리눅스는 이 칸들을 블록 디바이스로 부릅니다

HDD

Hard Disk Drive · 회전 원판

  • 모터로 도는 자기 디스크 + 헤드
  • 큰 용량을 싸게 — 보관·백업용
  • 속도는 느리고 충격에 약함

SSD / NVMe

Solid State · 플래시 메모리

  • 움직이는 부품 없음 → 빠르고 조용
  • OS·앱·DB가 사는 주 디스크
  • NVMe는 PCIe로 직결 — 가장 빠름

USB · SD

이동식 저장장치

  • 꽂으면 인식·뽑으면 사라지는 칸
  • 설치 USB · 사진 백업용
  • 리눅스 입장에선 /dev/sdb 같은 디스크

블록 디바이스(Block Device) = 데이터를 일정한 덩어리(블록) 단위로 읽고 쓰는 저장장치. 리눅스에서는 모두 /dev/... 라는 파일처럼 보입니다 — "디스크도 파일이다."

# CONCEPT · 02

디바이스 이름 규칙 — /dev/sda1은 어떻게 읽나?

이름만 봐도 "어떤 디스크의 몇 번째 파티션"인지 즉시 알 수 있도록

SCSI/SATA·USB 계열

/dev/sd[a-z][숫자]

/dev/sda1
경로 SCSI/SATA 디스크 순서 파티션 #
이름의미
/dev/sda첫 번째 디스크 (전체)
/dev/sda1첫 번째 디스크의 1번 파티션
/dev/sdb두 번째 디스크 — 새로 꽂으면 보통 이것
/dev/sdb2두 번째 디스크의 2번 파티션

부팅 순서·USB 꽂은 타이밍에 따라 sda↔sdb가 바뀔 수 있음 → 영구 설정에는 다음 슬라이드의 UUID 사용을 추천.

NVMe(M.2) 계열

/dev/nvme[N]n[M]p[K]

/dev/nvme0n1p1
경로 컨트롤러 네임스페이스 파티션
이름의미
/dev/nvme0n10번 NVMe 컨트롤러의 1번 네임스페이스(=디스크)
/dev/nvme0n1p1그 디스크의 1번 파티션 — "p"가 붙는 점에 주의
/dev/nvme1n12번째 NVMe 컨트롤러

기억법: SATA는 sda1(파티션 번호 바로), NVMe는 nvme0n1p1(p 다음 번호).

# CONCEPT · 03

df vs du — 같은 "용량"이지만 보는 시점이 다르다

"방 전체에 남은 자리" vs "이 폴더가 차지하는 자리" — 헷갈리면 평생 헷갈립니다

df (disk free)
"방(파일시스템) 전체 잔여공간"
파일시스템 단위로 전체 / 사용 / 남은 / 사용률을 보여줍니다. "디스크가 거의 다 찼나?"를 확인하는 첫 번째 명령.
$ df -h
Filesystem Size Used Avail Use% Mounted on
/dev/sda1 20G 6.4G 13G 34% /
tmpfs 1.9G 24K 1.9G 1% /run
# -h = human readable (K/M/G로 보기 좋게)
du (disk usage)
"이 폴더가 차지한 용량"
지정한 디렉터리 트리를 훑으며 합계를 알려줍니다. "누가 용량을 잡아먹나?"를 찾는 도구.
$ du -sh /var/log
523M /var/log
$ du -sh *
12M Documents
2.4G Downloads
9.1G Videos
# -s = summary(합계만), -h = human readable
구분dfdu
보는 시점파일시스템(마운트) 전체지정한 디렉터리 트리
속도매우 빠름 (메타데이터만 읽음)느릴 수 있음 (실제로 디렉터리를 훑음)

파일시스템 = "디스크의 책장 정리법"

ext4

리눅스의 "기본 책장"

  • 거의 모든 리눅스 배포판의 기본값
  • 안정성·호환성이 검증된 대중 표준
  • 저널링(작업 일지)으로 갑작스런 종료에도 강함

xfs

큰 파일·서버 친화

  • 대용량 파일·디렉터리에 강함
  • RHEL/Rocky 등 서버에서 자주 기본값
  • 한 번 만들면 크기 축소는 불가 — 확장만

vfat / exFAT

"모든 OS가 읽는 책장"

  • USB·SD카드의 단골 — 윈도우·맥과 호환
  • 리눅스 고유 권한·심볼릭링크 약함
  • EFI 부팅 파티션도 vfat 계열

파일시스템(File System) = 디스크 안에 "파일·디렉터리·권한"을 어떻게 배치할지 정한 규약. 이번 장에서는 ext4를 기본으로, 필요할 때만 vfat·xfs를 만나봅니다.

# CONCEPT · 02

마운트(mount) — "디렉터리에 디스크를 꽂는다"

디스크는 "칸", 디렉터리는 "칸을 들여다보는 창문". 마운트는 둘을 잇는 작업

마운트 옵션 맛보기 — ro·noexec·nosuid

"꽂는 방식"을 옵션으로 조정 — 보안·실수 방지의 첫 칸막이

ro
read-only · 읽기 전용
쓰기 시도 시 즉시 Read-only file system 오류.
  • • 백업용·증거 디스크 — 절대 안 바뀌어야 할 때
  • • /boot를 ro로 둬서 실수 차단
$ sudo mount -o ro /dev/sdb1 /mnt/x
$ echo a > /mnt/x/a.txt
bash: a.txt: Read-only file system
noexec
실행 차단
이 마운트 안의 실행 파일을 실행 못 함. 읽고 쓰는 건 OK.
  • • 일반 사용자 업로드 폴더(/tmp·/home)에 적용해 악성 스크립트 실행 차단
  • • 단, 같은 파일을 다른 위치로 옮기면 실행 가능
$ sudo mount -o noexec /dev/sdb2 /mnt/y
$ /mnt/y/hello.sh
bash: ./hello.sh: Permission denied
nosuid
SetUID 비활성
SetUID 비트가 있는 파일도 루트 권한으로 승격되지 않음.
  • • USB·외부 파일시스템에서 권한 상승 공격 차단
  • • passwd 같은 SetUID 도구는 영향 X (시스템 마운트는 노옵션)
$ sudo mount -o nosuid /dev/sdb3 /mnt/z
# /mnt/z에 있는 SetUID 바이너리도 일반 권한으로만 실행

# SCENARIO 2 감사용 로그 디스크를 ro로 마운트해 "수정 불가" 증거로 보관.

④ 또는 새 디스크는 별도 마운트로 두고 데이터 옮기기 (애플리케이션 설정 다 손봐야 함)

물리 파티션은 시작·끝 섹터가 정해져 있어, 옆 칸을 침범하지 않고는 늘리지 못합니다.

LVM의 해법
"디스크를 풀(pool)로 본다"

① 디스크들을 VG(볼륨 그룹)라는 큰 풀로 묶음

② 그 풀에서 LV(논리 볼륨)를 잘라서 사용

③ /var가 가득 차면 → lvextend +1G → 즉시 1G 추가

④ 풀이 모자라면 새 디스크를 vgextend로 풀에 추가

"서비스 중단 없이 디스크가 자란다" — 운영자가 LVM을 사랑하는 이유.

LVM의 아이디어 = "파티션 대신 한 단계 추상화를 더 둔다." 물리 디스크와 사용자 사이에 유연한 층을 끼워 넣는 것.

절대값 vs 상대값 vs 퍼센트

메모리 보기 — free -h, "캐시는 낭비가 아니다"

"메모리가 90% 찼다"는 메시지에 놀라지 마세요 — 리눅스는 남는 메모리를 캐시로 활용합니다

Total: 4.0 GiB 각 영역의 크기
used 0.9G
buff
cache 1.5G
available 2.5G
used : 실제 앱이 사용 중
buffers : 디스크 쓰기 대기
cache : 디스크 읽기 캐시
available : "정말 남은 양"
free -h 출력 읽기

total ≠ used + free

$ free -h
total used free shared buff/cache available
Mem: 3.8Gi 884Mi 1.2Gi 24Mi 1.7Gi 2.5Gi
Swap: 2.0Gi 0B 2.0Gi

중요한 건 available 칼럼. "지금 새 프로세스가 쓸 수 있는 메모리"를 의미합니다. free가 0이라도 available이 충분하면 OK.

왜 캐시는 좋은가

disk >>> memory (수천 배 차이)

리눅스는 남는 메모리를 읽기 캐시로 채워둡니다. "또 읽어달라"는 요청이 오면 디스크가 아닌 메모리에서 즉시 응답 → 속도 향상.

앱이 메모리를 더 요구하면 캐시는 즉시 비워져 자리를 내줍니다. "메모리가 차 있다 ≠ 부족하다."

랜섬웨어 감염

# COMMAND · 03

rsync — 바뀐 것만 빠르게 복사

매번 전체를 복사하지 않고 달라진 파일만 전송 — 백업·미러링·동기화의 표준

❌ cp -r 만 쓰면

매번 모든 파일을 다시 읽고 다시 씀 → 10GB면 매번 10GB

✓ rsync는

날짜·크기·체크섬으로 비교 → 바뀐 파일만 전송 → 두 번째부터 수초

자주 쓰는 옵션 (-avh --delete)

"백업 5종 세트"

옵션의미
-aarchive — 권한·심볼릭링크·날짜 보존(=-rlptgoD)
-vverbose — 진행 출력
-hhuman readable — 사람 보기 좋은 단위
-ndry-run — 실제로 안 옮기고 시뮬레이션 (필수!)
-z전송 시 압축 (네트워크 백업)
--delete원본에 없는 파일은 대상에서도 삭제
--exclude=PATTERN제외 패턴 (예: '*.tmp')
사용 패턴

로컬 · 원격 모두 같은 문법

# 1) 시뮬레이션 (-n 필수)
$ rsync -avhn /home/student/ /mnt/backup/student/
# 2) 실제 실행
$ rsync -avh /home/student/ /mnt/backup/student/
# 3) 원본과 완전히 일치(원본 없는 건 삭제)
$ rsync -avh --delete /home/student/ /mnt/backup/student/
# 4) SSH로 원격
$ rsync -avhz /home/student/ user@server:/backup/student/

슬래시 주의: /home/student/(끝에 /)는 "안의 것을", /home/student(끝 / 없음)은 "그 폴더 자체를" 복사.

미니 프로젝트 — 일일 자동 백업 스크립트 + crontab 등록

앞 6개 슬라이드 내용을 모아 "매일 새벽 3시 백업" 시스템을 완성합니다

/usr/local/bin/daily-backup.sh
스크립트 — tar + rsync + 보관기간 7일
1#!/bin/bash
2# Daily backup: /etc snapshot + /home rsync mirror
3set -euo pipefail
4
5TODAY="$(date +%F)"
6DEST="/var/backups"
7KEEP_DAYS=7
8
9mkdir -p "$DEST/etc" "$DEST/home"
10
11# 1) /etc snapshot (tar.gz, 매일 파일)
12tar -czf "$DEST/etc/etc-$TODAY.tar.gz" /etc 2>/dev/null
13
14# 2) /home rsync mirror
15rsync -avh --delete /home/ "$DEST/home/"
16
17# 3) 오래된 etc 백업 청소 (7일 초과)
18find "$DEST/etc" -name "etc-*.tar.gz" -mtime +$KEEP_DAYS -delete
19
20echo "[$(date '+%F %T')] backup done"
설치 단계 (5스텝)
스크립트 작성 → 권한 → 테스트 → cron 등록 → 검증
  • 1 스크립트 만들기
    sudo nano /usr/local/bin/daily-backup.sh → 좌측 내용 붙여넣기
  • 2 실행 권한 부여
    sudo chmod +x /usr/local/bin/daily-backup.sh
  • 3 수동 테스트 (cron 등록 전 필수)
    sudo /usr/local/bin/daily-backup.sh
  • 4 cron 등록 — 매일 새벽 3시 실행
    sudo crontab -e 후 한 줄 추가:
0 3 * * * /usr/local/bin/daily-backup.sh >> /var/log/backup.log 2>&1
